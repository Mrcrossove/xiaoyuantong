# 校园通商用上线缺口与开发推进计划

## 1. 文档目的

本文件用于统一整理当前项目距离“真正上线商用”仍缺失的功能、配置、工程能力与合规能力，并给出后续开发推进顺序。

当前项目已部署到云服务器，后续开发发布流程约定如下：

1. 先在本地完成代码修改与验证。
2. 将本地变更推送到 GitHub。
3. 在云服务器执行 `git pull` 拉取最新代码。
4. 按服务端、管理后台、商家后台分别构建与重启。

## 2. 当前项目状态判断

当前项目具备以下基础：

- 小程序端、管理后台、商家后台、后端 API 的基本结构已存在。
- 校园认证、帖子、店铺、商品、订单、退款、钱包、提现、后台权限等基础业务已经有雏形。
- 云服务器、Nginx、PM2、前后端构建脚本已经有初步部署方式。

但当前整体更接近：

- 可演示
- 可联调
- 可做小范围测试

还不适合直接作为正式收费商用系统对外开放。

## 3. 上线前必须补齐的缺失项

### 3.1 登录与账号安全

- 管理员账号密码当前仍是明文存储与明文校验。
- 商家账号密码当前仍是明文存储与明文校验。
- 默认密钥与默认管理员账号仍存在。
- Token 体系只有基础签名，没有刷新机制、吊销机制、黑名单机制、设备踢下线机制。
- 商家验证码登录缺少频控、防刷、人机校验。

必须补齐：

- 密码改为哈希存储。
- 默认账号与默认密码全部移除。
- 生产环境强制要求高强度密钥。
- 增加登录失败限制、验证码发送限制、IP 频控。
- 增加会话失效与异常登录控制。

### 3.2 微信登录与小程序身份链路

- 小程序登录目前仍支持 mock 逻辑。
- 小程序也支持仅凭 deviceId 建立用户，真实商用风险较高。
- 当前缺少完整的微信正式登录生产配置。

必须补齐：

- 关闭微信 mock。
- 强制走真实微信登录链路。
- 明确用户唯一身份策略：openid / unionid / deviceId 的使用原则。
- 明确老账号迁移策略。

### 3.3 微信支付正式闭环

- 当前支付仍支持 mock。
- 当前支付确认依赖客户端主动确认，不是完整的支付回调闭环。
- 未看到独立的支付回调验签、异步补单、重复通知处理。
- 支付成功后的账务确认与状态流转还不够稳健。

必须补齐：

- 接入正式微信支付参数。
- 增加支付回调接口。
- 增加回调验签与重复回调幂等处理。
- 增加支付结果补单机制。
- 增加订单支付日志与支付异常告警。

### 3.4 退款正式闭环

- 退款流程虽然已有业务逻辑，但需要以正式微信退款能力为准。
- 缺少退款回调、退款状态异步确认、异常退款重试、失败记录。

必须补齐：

- 接入正式微信退款。
- 增加退款结果确认机制。
- 增加退款失败补偿与人工介入流程。
- 增加退款流水记录与对账字段。

### 3.5 提现打款能力

- 当前提现审核通过后，主要是改数据库余额，不等于真实打款。
- 未形成真正的微信零钱/银行卡打款闭环。
- 没有打款单号、打款回执、失败重试、到账确认。

必须补齐：

- 确定提现打款方式。
- 实现实际打款接口。
- 增加打款状态跟踪。
- 增加失败重试、人工复核、回滚策略。
- 增加提现对账能力。

### 3.6 风控与内容安全

- 风控当前仍偏 mock 形态。
- 真实生产模式下风控逻辑需要重新梳理。
- 当前仅有简单关键词与频率限制，不足以商用。

必须补齐：

- 明确生产环境风控启用策略。
- 增加内容敏感词、导流识别、诈骗识别。
- 增加验证码、频控、黑名单、封禁策略。
- 增加举报处理闭环与用户申诉机制。

### 3.7 文件上传与资源存储

- 当前图片上传是本地磁盘直写。
- 线上缺少对象存储、CDN、资源生命周期策略。
- 未看到文件类型安全校验、内容审核、异常文件处理。

必须补齐：

- 将上传迁移到云对象存储。
- 接入 CDN。
- 增加文件大小、类型、扩展名、内容校验。
- 明确资源目录规划、清理策略、备份策略。

### 3.8 域名、HTTPS 与正式访问配置

- 当前小程序接口地址有硬编码 IP。
- Nginx 配置仍是基础版本。
- 未看到正式 HTTPS、证书续期、80 到 443 跳转配置。

必须补齐：

- 配置正式域名。
- 所有前后端访问改为正式域名。
- 完成 HTTPS 证书配置。
- 小程序合法域名、上传域名、下载域名全部切换。

## 11. 当前版本部署前必须确认

由于后端已经加入生产环境安全校验，如果服务器使用 `NODE_ENV=production`，那么以下配置不满足时，API 启动会直接失败：

- `APP_SECRET` 不能是默认值、占位值或弱值。
- `WECHAT_USE_MOCK` 必须为 `false`。
- `PAY_USE_MOCK` 必须为 `false`。
- `SMS_PROVIDER` 不能为 `mock`。
- `PUBLIC_BASE_URL` 必须是 `https://` 开头的正式公网地址。

结合你当前已提供的信息，现阶段有几个现实限制：

- 生产域名暂时没有，所以 `PUBLIC_BASE_URL` 还无法切到正式 HTTPS 域名。
- 小程序 `appsecret` 还未提供，因此真实微信登录联调还不能完全打通。
- 短信服务未开通，因此验证码正式发送仍不能进入真实生产闭环。
- 支付相关信息尚未提供，因此支付/退款/分账仍属于下一阶段工作。

因此，当前最稳妥的推进方式是：

1. 先把本轮阶段 A 收口代码提交到 `main`。
2. 服务器拉代码前，先核对并修改 `admin-api` 的生产环境变量。
3. 如果你当前线上还是 mock 配置，就不要直接以新的生产校验配置重启正式 API。
4. 先准备域名、HTTPS、微信 `appsecret`、短信服务，再进入阶段 B 联调。
