# 校园平台小程序开发规则

## 1. 文档目的

本项目用于开发一个校园平台小程序，聚合校园生活相关服务。  
从本规则文档创建起，后续所有设计复刻、前端开发、方案记录、代码评审、验收检查、后端评估，均默认遵守本文件。

## 2. 总体开发原则

1. 以你提供的对标页面截图为准，优先保证页面结构、视觉层级、排版、间距、颜色、组件状态的高还原。
2. 默认先完成前端页面复刻，再逐步补齐交互、数据接入、状态管理和联调能力。
3. 当前阶段若页面数据来源未明确，先使用结构化 mock 数据实现，但必须保留后续替换为真实接口的清晰边界。
4. 所有新增代码必须可维护、可复用，避免一次性硬编码堆砌。
5. 每完成一次编码任务，必须进行自检、review 和可验收性确认。
6. 当前所有任务均以“可继续迭代”为前提，代码结构要为后续批量页面开发和后端接入预留空间。

## 3. 页面开发模式

后续你每次提供一个页面截图，我将按以下模式执行：

1. 先分析截图内容，拆解页面模块、组件、布局、样式层级和交互预期。
2. 在开始写代码前，先创建该任务对应的“子工程记录文件”。
3. 先输出并固化该任务的技术方案和实现方案，再开始编码。
4. 完成页面开发后，进行代码 review、错误排查、构建/运行检查和验收说明。
5. 明确记录本次任务已完成内容、未完成内容、风险点和后续建议。

## 4. 子工程记录文件规则

### 4.1 创建时机

每次开始一个新的编码任务之前，必须先创建对应的任务记录文件，然后才能开始写业务代码。

### 4.2 存放位置

统一放在根目录下：

`/tasks/`

如果目录不存在，先创建该目录。

### 4.3 命名规则

每次任务新建一个独立 markdown 文件，命名格式如下：

`tasks/YYYY-MM-DD-任务名称.md`

示例：

`tasks/2026-04-04-首页改版.md`

### 4.4 必填内容

每个任务记录文件至少包含以下内容：

1. 任务名称
2. 任务目标
3. 设计参考说明
4. 页面模块拆解
5. 技术方案
6. 实现方案
7. 涉及文件
8. 自检清单
9. 验收结果
10. 风险与后续建议

### 4.5 推荐模板

后续默认按以下模板创建：

```md
# 任务名称

## 任务目标

## 设计参考说明

## 页面模块拆解

## 技术方案

## 实现方案

## 涉及文件

## 自检清单

## 验收结果

## 风险与后续建议
```

## 5. 前端实现要求

1. 优先保证页面视觉还原度，其次保证组件复用性和代码结构清晰。
2. 样式实现需避免无意义的重复代码，公共样式、公共组件、主题变量应适度抽离。
3. 对截图中不明确的交互、状态、文案和动态数据，必须在任务记录文件中标注“按合理假设实现”。
4. 对页面中的列表、卡片、导航、服务入口、标签、弹层等模块，优先按组件化方式开发。
5. 每次开发时，尽量避免写死强耦合逻辑，便于后续对接后端接口。
6. 若项目已有设计风格、组件体系或目录规范，新增页面必须与现有项目保持一致。
7. 若截图存在响应式差异需求，默认同时兼顾常见手机尺寸下的显示效果。

## 6. 每次编码后的强制检查

每完成一次编码任务，必须执行以下检查：

1. 检查新增和修改代码是否存在语法错误。
2. 检查页面结构、样式和截图目标是否存在明显偏差。
3. 检查是否存在明显的命名混乱、重复代码、无效代码和潜在报错点。
4. 检查 mock 数据、组件参数、条件渲染、列表渲染是否合理。
5. 检查本次任务记录文件是否已补全“自检清单”和“验收结果”。
6. 若项目支持构建、运行或 lint/test，则尽量执行并记录结果。

## 7. Review 与验收规则

每次完成任务后，我都必须输出 review 结论，至少覆盖以下内容：

1. 本次修改是否发现报错。
2. 是否已完成基础自检。
3. 是否达到可验收状态。
4. 若未达到可验收状态，明确原因。
5. 若存在风险项，明确列出风险和建议处理方式。

验收说明至少包括：

1. 已完成内容
2. 待补充内容
3. 已知问题
4. 建议下一步

## 8. 后端评估阶段规则

当前阶段以后端开发为后置任务。  
待所有前端页面开发完成后，再统一进入后端评估与设计阶段。

届时我需要输出至少以下内容：

1. 数据库表结构设计建议
2. 核心业务实体和关系梳理
3. 接口清单建议
4. 后端服务模块拆分建议
5. 权限体系建议
6. 文件上传、消息通知、搜索、推荐等扩展能力建议
7. 技术选型建议
8. 分阶段开发方案

## 9. 沟通与执行约定

1. 你后续提供页面截图后，我默认进入该页面的分析和复刻流程。
2. 在没有特殊说明的情况下，我会直接按本规则创建任务记录文件并开始实现。
3. 如果截图信息不足以支撑完整实现，我会基于合理假设先完成可落地版本，并在任务文档中注明。
4. 如果发现现有项目结构不利于继续开发，我会先说明调整原因，再进行必要整理。

## 10. 本文档的约束范围

本规则文档对当前目录下后续全部开发任务生效。  
除非你明确要求修改，否则我后续默认持续遵守本文件。
